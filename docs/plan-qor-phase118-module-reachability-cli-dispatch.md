# Plan: Module reachability — `qor-logic reliability`/`scripts` CLI dispatch (Option A)

**change_class**: feature

**doc_tier**: standard

**terms_introduced**:
- term: module dispatch subcommand
  home: qor/references/doctrine-governance-enforcement.md

**boundaries**:
- limitations: The dispatch resolves modules through the CLI's own `sys.executable`; it does not add a module allowlist or registry. Any importable `qor.reliability.<mod>` / `qor.scripts.<mod>` is dispatchable, and unknown modules surface Python's native "No module named" error with a non-zero exit.
- non_goals: Removing the bare `python -m qor.X` form (retained as the in-venv fallback per the hybrid strategy); reinstalling or upgrading the operator's global `qor-logic`; fixing the pre-existing qor-research install drift.
- exclusions: Non-canonical skill copies under `.claude/skills/` (installed artifacts, regenerated by `qor-logic install`) and `qor/dist/variants/` (regenerated by `dist_compile`, not hand-edited).

## Open Questions

None. Invocation strategy resolved up-front: **hybrid** — add the dispatch, migrate skill invocation lines to `qor-logic reliability/scripts <mod>`, retain `python -m qor.X` as the documented in-venv fallback, and update Phase 90 + wiring + §138 tests to accept either form.

## Context

GH #150 (umbrella #147, follow-on to #79/#38). Skills call `python -m qor.reliability.*` / `python -m qor.scripts.*` from arbitrary shells, but bare `python` only resolves the module when the install venv is active. The fix (Option A): a self-resolving CLI dispatch that invokes the target module via the CLI process's own `sys.executable`, which is correct regardless of which `python` is on PATH.

### Changes

In `qor/cli.py`:

```python
import subprocess
import sys

def _register_module_dispatch(sub) -> None:
    for family in ("reliability", "scripts"):
        sp = sub.add_parser(
            family,
            help=f"run a qor.{family}.<module> via the CLI's own interpreter",
        )
        sp.add_argument("module", help=f"module name under qor.{family}")
        sp.add_argument(
            "args", nargs=argparse.REMAINDER,
            help="arguments passed through to the module verbatim",
        )

def _do_module_dispatch(family: str, args: argparse.Namespace) -> int:
    target = f"qor.{family}.{args.module}"
    completed = subprocess.run([sys.executable, "-m", target, *args.args])
    return completed.returncode
```

- Call `_register_module_dispatch(sub)` inside `_build_parser`.
- In `_dispatch`, add `"reliability"` / `"scripts"` keys mapping to `lambda: _do_module_dispatch(args.command, args)`.

`sys.executable` is the interpreter running the CLI. When `qor-logic` is a pipx/global install, its console-script shim hardcodes that interpreter, so `qor-logic reliability skill_admission qor-plan` resolves `qor.reliability.skill_admission` regardless of the active shell venv. Streams are inherited (no capture) so module stdout/stderr pass straight through. `argparse.REMAINDER` captures all trailing tokens verbatim, so module flags (`--session`, `--ledger`, ...) are not interpreted by the dispatch parser.

De-complecting note: "which interpreter runs the module" is separated from "which python is on PATH." No allowlist is introduced — `-m qor.{family}.{module}` resolution is the single source of truth; the f-string prefix confines targets to the two packages, and the list-form `subprocess.run` (no `shell=True`) admits no shell injection.

### Unit Tests

- `tests/test_cli_module_dispatch.py::test_reliability_dispatch_resolves_with_venv_inactive` - invokes `[sys.executable, "-m", "qor.cli", "reliability", "skill_admission", "qor-plan"]` in a subprocess whose env has `VIRTUAL_ENV` removed and venv `Scripts`/`bin` dirs stripped from `PATH`; asserts exit 0 and `ADMITTED` in stdout. Confirms resolution does not depend on an active venv. Invokes `qor.cli` (code under test), never the PATH `qor-logic` (which may be a stale global).
- `::test_scripts_dispatch_arg_passthrough` - invokes `qor.cli scripts active_phase --help` (or a known no-op flag) and asserts the module received the args (module-specific marker in output), proving REMAINDER passthrough.
- `::test_dispatch_propagates_nonzero_exit` - dispatches a module invocation known to exit non-zero (e.g. `reliability skill_admission` with an unadmitted skill name) and asserts the dispatch returns that same non-zero code, not 0.
- `::test_unknown_module_returns_nonzero` - `reliability no_such_module` returns non-zero (Python "No module named") rather than crashing the CLI.
- `::test_family_prefix_isolation` - `reliability active_phase` (a `qor.scripts` module, absent from `qor.reliability`) fails to resolve, proving the family prefix is not bypassable.
