## Razor Exemption (D-107.2; declared upfront for audit transparency)

The `requirements-sbom.txt` file produced by `pip-compile --generate-hashes` against `cyclonedx-bom` weighs ~390 lines (lxml alone carries ~120 wheel hashes across its supported platforms). This exceeds the Section 4 Razor 250-line file budget.

**Exemption rationale**: the Razor 250-line file budget applies to human-authored source code, where line count tracks authoring complexity and cognitive load on maintainers. Deterministic generated lockfiles do not exhibit that property: their size is a function of the dependency graph and platform wheel coverage, both decided by upstream packages, not by qor-logic authors. The file is regenerated mechanically via `pip-compile --generate-hashes --output-file=requirements-sbom.txt requirements-sbom.in` and reviewed for cryptographic correctness (each hash is verified at install time by pip's `--require-hashes`), not for readability.

**Exemption scope**: this exemption applies only to `requirements-sbom.txt`. The existing `requirements-release.txt` (Phase 102, ~21 lines) also benefits from the same rationale but does not need the exemption because it fits the budget naturally. No human-authored file gains an exemption.

**Exemption authority**: declared at plan-time and approved at audit-time per `qor/references/doctrine-shadow-genome-countermeasures.md` Section 4 self-application discipline. The audit's Razor pass MUST cite this exemption block when evaluating `requirements-sbom.txt`.

## Design notes

### D-107.1 — Cooling-period lint flip

Single-line change in `.github/workflows/pr-dependency-review.yml`: remove the `|| true` wrap from the lint step. The lint already supports the three-way exit code (0 clean / 1 violation / 2 network), and the workflow's natural failure-on-non-zero behavior is the desired enforcement.

V1 / V1.1 visibility-stage operator-evidence: the operator (`@Knapp-Kevin`) is the only PR author on qor-logic; the V1.1 cooling-period lint has now run on Phase 105/106 PRs without false-positive WARN; the surface is operator-controlled and the override procedure (META_LEDGER entry + `dep-admit-override` PR label) is in place. The operator-directed flip authority comes from this `/qor-auto-dev-1` invocation.

### D-107.2 — `cyclonedx-bom` hash-pinning

`requirements-sbom.in` declares the single top-level dep `cyclonedx-bom`. `requirements-sbom.txt` is generated via `pip-compile --generate-hashes`; commit both. Release workflow build job:

```yaml
- name: Install SBOM tool (hash-pinned) and generate SBOM
  run: |
    pip install --require-hashes -r requirements-sbom.txt
    cyclonedx-py environment --of JSON --output-file dist/sbom.json
```

Replaces the current `pip install cyclonedx-bom && cyclonedx-py environment ...` two-line block.

### D-107.3 — Range-pin coverage (lower-bound check)

New `parse_pyproject_range_pins(text) -> list[LockfileEntry]` in `qor/scripts/_dep_admit_common.py`. Returns the lower-bound version for entries matching `>=X.Y.Z` or `~=X.Y.Z` PEP 440 forms. Range pins with no lower bound (`<X.Y.Z`) and unbounded specifiers are excluded. The returned `LockfileEntry` uses the lower-bound version as `version`.

Rationale for lower-bound: pip's installer resolves to the latest version satisfying the constraint. The earliest version a fresh install COULD resolve to is the lower bound. Checking the lower-bound's upload time is the conservative interpretation: if THAT version is older than 14 days, every version pip might pick is also older; if it's within the window, there's a risk pip picks a within-window version. Catches the "freshness vector" admission risk precisely.

Lint integration: `run_lint()` gains the union with the existing exact-pin diff. The lint's bumps list includes range-pin bumps with the lower-bound version. The pyproject diff function `diff_lockfile_against_base` works unchanged because `LockfileEntry` already supports the shape.

### D-107.4 — GH-issue emit for tracker

`qor/scripts/dep_admit_override_tracker.py` gains `--emit-issue` flag. When set:

1. Compute the markdown table (existing logic).
2. Filter to `--filter due` (overrides ≥30 days old).
3. If no due entries, exit 0 silently.
4. If due entries exist, build a rollup issue body (markdown table + summary line) and invoke `gh issue create --title "..." --body "..." --label dep-admit-override-review`.
5. Print the new issue URL to stdout.

Single rollup per invocation (not per entry) avoids spam; operator-controlled cadence. The `dep-admit-override-review` label is documented in the Phase 103 doctrine for traceability.

Network behavior: gh CLI shell-out (consistent with Phase 106 PR-label query); fails fast on non-zero (no automatic retry since this is an operator-initiated action). `--emit-issue` without GH context (no `GITHUB_TOKEN` available) errors loudly via gh's own validation.

### D-107.5 — CODEOWNERS solo-owner doctrine note

`qor/references/doctrine-governance-enforcement.md` gains a new §6.1 "CODEOWNERS operational mode" subsection. Documents that the project intentionally operates in solo-owner mode (`@Knapp-Kevin` for all security-critical files) and lists the trigger conditions for expanding the reviewer pool: (1) a second maintainer joins, (2) the project federates with other Qor-logic deployments, (3) compliance audit requires reviewer separation, OR (4) operator-initiated formalization of a maintainer team. Until one of these triggers fires, the solo-owner state is the chosen operational mode, not a TODO.

This closes the carry-forward via documentation — no scaffold change. Future expansion is gated on a real trigger, not a generic deferral.
