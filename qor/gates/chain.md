# Qor Gate Chain

**Version**: 1 (Phase 3)
**Enforcement**: Advisory (soft). Missing/invalid prior artifact surfaces a prompt; user override logs a `gate_override` event (severity 1) in `docs/PROCESS_SHADOW_GENOME.md`.

## Sequence

```
(ideate?) -> research -> plan -> audit -> implement -> substantiate -> validate -> (remediate?)
```

**Phase 59**: `ideate` is an optional pre-research phase (advisory). When `ideation.json` exists in the session dir, `/qor-research` and `/qor-plan` accept it as a valid prior artifact. Hotfixes MAY skip ideation. `remediate` is conditional ‚Äî triggered by threshold breach in the Process Shadow Genome, or by explicit `/qor-remediate` invocation. Not part of the strict forward chain.

## Artifact locations

Runtime: `.qor/gates/<session_id>/<phase>-iter<N>.json` (immutable, one per emission) plus `.qor/gates/<session_id>/<phase>.json` (latest copy; see below).

`session_id` format: `<YYYY-MM-DDTHHMM>-<6hex>` (e.g. `2026-04-15T1743-a3f9c2`). No colons ‚Äî safe as a directory name on Windows.

Generated by `qor/scripts/session.py get_or_create`. Cached in `.qor/current_session` file, regenerated if older than 24h.

### Iteration versioning (Phase 173; GH #237)

Multi-iteration phases are the normal path (audit VETO‚ÜíPASS, substantiate FAIL‚ÜíSEAL). Every emission through `write_gate_artifact` lands in its own `<phase>-iter<N>.json` (N = highest existing iteration + 1, starting at 1); the writer never re-targets an existing iteration, so evidence already bound by a seal can never be overwritten by a later run. The unversioned `<phase>.json` is refreshed on each write as a byte-identical latest copy ‚Äî every pre-173 consumer of the singleton path keeps working unchanged.

- Resolution (`gate_chain.check_prior_artifact`, `read_phase_artifact`) selects the highest iteration; session dirs with no `-iter` files (pre-173) fall back to the singleton. No retroactive migration.
- Provenance sidecars exist for both forms: `<phase>-iter<N>.provenance` binds the immutable file; `<phase>.provenance` is refreshed beside the singleton for committed-artifact verification (`gate_provenance verify-committed`) and evidence reconstruction.
- `audit_history.jsonl` rows carry `artifact_filename` naming the versioned file each audit emission landed in (absent on pre-173 rows).
- The gate-written hook event's `artifact_path` / `payload_sha256` reference the versioned file.

## Risk-tiered gate depth (Phase 168; GH #248)

The plan artifact may declare `required_gate_artifacts` (Shape 3 per research entry #394). The short chain `[plan, implement, substantiate]` -- omitting ONLY audit -- is permitted exclusively for L1-risk (per `qor/capabilities/risk.py` routing over the plan's affected files) `hotfix` changes; the schema forbids `feature`/`breaking` from omitting audit, `qor/scripts/tier_guard.py` verifies the declaration at implement time, and `gate_chain_completeness` + `gate_provenance verify-committed` honor it at seal/CI/merge (illegal or absent declarations resolve to the full set, fail-closed). Declaring the short chain emits a severity-1 `gate_override` shadow event -- never silent. Absent field == full chain: all pre-168 sessions grandfathered by construction.

## Skill integration (Phase 52 wiring)

Skills add to their YAML frontmatter:

```yaml
phase: plan
gate_reads: research
gate_writes: plan
```

At startup, skills call:

```python
from gate_chain import check_prior_artifact, emit_gate_override
result = check_prior_artifact(current_phase="plan")
if not result.found:
    # prompt user; if override, emit event
    emit_gate_override(current_phase="plan", prior_phase="research",
                       reason="...", session_id=session_id)
```

Skills write their output artifact near end-of-protocol:

```python
from validate_gate_artifact import write_artifact
write_artifact(phase="plan", data={...})
```

## Override semantics

Override is permitted for any gate. Each override:
- Logs a `gate_override` event (severity 1) with `details` capturing reason, prior_phase, current_phase
- Counts toward the Process Shadow Genome threshold (10 severity points -> issue creation)
- Is visible in `docs/PROCESS_SHADOW_GENOME.md` JSONL and via `qor/scripts/check_shadow_threshold.py`

Repeated overrides on the same gate indicate either a process defect (gate too strict) or a systemic shortfall (skill not producing artifacts). `/qor-remediate` evaluates patterns.

## Non-forward paths

- **Remediate loop**: `remediate` consumes the shadow log directly, not a prior gate artifact. It can be invoked after any phase.
- **Audit VETO**: if `audit.json.verdict == "VETO"`, `implement` should not proceed. Skill-layer enforcement (Phase 52 wiring): `QOR_SKILL_ACTIVE` env-var provenance binding in `gate_chain.write_gate_artifact` + `gate_chain_completeness.check()` invoked from `/qor-substantiate` Step 7.8 + pre-merge CI gate. Bypass surface closed.
- **Debug interrupt**: if regression/hallucination/degradation is detected during `implement` or `substantiate`, `/qor-debug` is the legal next action (not `implement` again). This is a skill-level rule, not a gate schema rule.
